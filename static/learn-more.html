How Our Fraud Detection System Works

Our system is designed to detect **real payment fraud in real time** using **actual user behavior**, not fake inputs or manual selections. The goal is to simulate how modern payment platforms like UPI, Stripe, or card networks protect transactions.

Core Idea

Fraud rarely happens in isolation. It leaves signals — device changes, IP changes, unusual timing, and rapid transaction patterns. Our platform captures these signals automatically and feeds them into a trained Machine Learning model.

No manual selection.
IP address, device information, session timing, and transaction frequency are captured automatically at payment time.

End-to-End Flow

1. User Authentication

Users register and log in using Firebase Authentication. Each user has a unique identity that persists across sessions.

2. Real-Time Data Capture

When a payment is attempted, the backend automatically captures: IP address, device fingerprint (User-Agent), timestamp, and session activity.

3. Risk Signal Generation

The backend compares current session data with historical user data stored in the database to detect changes or anomalies.

4. Machine Learning Decision

A trained Random Forest model evaluates the risk signals and classifies the transaction as Normal or Fraud.

5. Action

Normal transactions are allowed instantly. Fraudulent transactions are blocked and explained to the user.

What Makes This Realistic

Unlike demo projects that let users choose “Suspicious IP” or “Bad Device”, our system works the way real payment systems do:

  • IP changes detected automatically
  • Device changes detected across sessions
  • Transaction frequency tracked per user
  • Time-based anomaly detection (late night transactions)
  • ML model isolated from raw identifiers

Why This Architecture Matters

Raw identifiers like IPs and device strings are unstable and high-cardinality. Our design converts them into stable, explainable risk features before passing them to the ML model — exactly how production fraud pipelines work.

Backend rules extract risk → ML decides → system acts.
This separation keeps the model explainable and scalable.

Who This Is For

This project is designed for real-world fintech scenarios: UPI apps, payment gateways, e-commerce checkout systems, and internal fraud monitoring dashboards.
